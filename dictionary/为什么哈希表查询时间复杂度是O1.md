# 使用 Hash 表结构，查询数据为什么时间复杂度能到 O(1)

## 问题

使用 hash 表结构，查询数据为什么时间复杂度能到 `O(1)`？

## 回答

哈希表查询之所以能到 `O(1)`，核心原因是：

- 它不是像数组遍历、链表逐个找那样比较很多次
- 而是先用哈希函数，把 `key` 直接映射成一个数组下标
- 然后一次定位到大致位置，再取值

可以理解成：

```text
key -> 哈希函数 -> 下标 index -> 直接访问数组[index]
```

数组按下标访问本来就是 `O(1)`，所以哈希表在平均情况下查询也接近 `O(1)`。

## 为什么快

- 哈希表底层通常是一个数组
- 查询一个 `key` 时，不需要扫描所有元素
- 只要把 `key` 经过哈希函数计算出位置，就能直接去对应的桶或槽位查找
- 这个“计算位置 + 访问数组”的过程是常数级的，所以平均时间复杂度是 `O(1)`

## 举个例子

假设有一个哈希函数：

```text
hash("apple") = 3
hash("banana") = 7
```

那么存储时：

- `"apple"` 放到数组下标 `3`
- `"banana"` 放到数组下标 `7`

查询 `"apple"` 时：

- 再算一次 `hash("apple")`
- 直接去下标 `3` 找

这样不需要从头到尾遍历，所以速度很快。

## 为什么说是平均 O(1)，不是绝对 O(1)

因为会有哈希冲突，也就是两个不同的 `key`，可能算出同一个下标：

```text
hash("apple") = 3
hash("grape") = 3
```

这时就不能只看一个元素了，需要在这个位置继续比较。

常见处理方式：

- 拉链法：一个位置挂一个链表或数组
- 开放寻址法：冲突后继续找下一个可用位置

所以：

- 平均情况：冲突少，查询接近 `O(1)`
- 最坏情况：如果很多 `key` 都冲突到一起，可能退化成 `O(n)`

## 为什么实际中通常还是很快

因为优秀的哈希表实现会尽量避免冲突：

- 哈希函数尽量让数据分布均匀
- 控制装载因子（元素个数 / 桶数量）
- 元素太多时自动扩容，然后重新哈希

这样大多数查询通常只需要：

- 1 次哈希计算
- 1 次数组访问
- 很少量冲突处理

所以平均复杂度就能保持在 `O(1)`。

## 一句话总结

哈希表查询快，是因为它把“查找元素”变成了“通过哈希函数直接定位数组下标”，省掉了大量遍历；因此平均时间复杂度是 `O(1)`，但如果冲突严重，最坏情况会退化到 `O(n)`。
