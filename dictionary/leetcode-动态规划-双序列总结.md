# LeetCode 动态规划-双序列总结

## 一、双序列动态规划的定义、理解和思维

### 1. 什么是双序列动态规划

- `双序列动态规划`，指的是题目同时给出两个有顺序的对象，通常是两个字符串或两个数组，然后要求我们研究它们之间的某种关系。
- 这里的“序列”强调的是`顺序`，比如字符串 `"abcde"`、数组 `[1, 3, 5, 7]` 都可以看成序列。
- 这类题的答案往往不是只由一个序列决定，而是由`两个序列之间的匹配关系、转换关系或统计关系`决定。

常见状态定义是：

```text
dp[i][j]
```

表示：

- 第一个序列前 `i` 个元素
- 和第二个序列前 `j` 个元素
- 在当前题目下的答案

### 2. 为什么它适合用动态规划

双序列问题通常满足动态规划的两个典型条件：

- `重复子问题`：很多前缀组合会被反复计算。
- `最优子结构`：大问题的答案，可以由更小规模的前缀问题推出。

例如：

- 求两个字符串的最长公共子序列时，会不断用到更短字符串前缀之间的答案。
- 求编辑距离时，当前状态也依赖更短前缀之间的转换结果。

所以这类题常常适合定义成一个二维表，通过`填表`的方式，从小问题逐步推出大问题。

### 3. 双序列动态规划的核心思维

做双序列动态规划时，建议固定按下面顺序思考：

#### 第一步：定义状态

先明确一句话：

```text
dp[i][j] 表示什么
```

最常见的表达方式是：

```text
dp[i][j] = 第一个序列前 i 个元素 和 第二个序列前 j 个元素 的某种答案
```

这个“答案”可能是：

- 最长长度
- 最少操作数
- 是否可行
- 方案数

#### 第二步：只看新增的末尾元素

因为：

- 第一个序列前 `i` 个，比前 `i - 1` 个，只多了一个元素
- 第二个序列前 `j` 个，比前 `j - 1` 个，只多了一个元素

所以研究 `dp[i][j]` 时，真正新增的信息通常只有：

- 第一个序列的末尾元素
- 第二个序列的末尾元素

于是思考重点会集中在：

- 这两个末尾元素是否相等
- 它们是否可以配对
- 如果不能同时用，应该舍弃哪一个

#### 第三步：分情况讨论转移

双序列动态规划最常见的转移来源有三个位置：

- `dp[i - 1][j - 1]`
- `dp[i - 1][j]`
- `dp[i][j - 1]`

它们分别对应的含义通常是：

- `左上`：两个末尾元素一起处理
- `上边`：舍弃第一个序列的末尾元素
- `左边`：舍弃第二个序列的末尾元素

#### 第四步：确定初始化

这类题一般会定义：

- `dp[0][j]`
- `dp[i][0]`

也就是让其中一个序列为空，看答案应该是什么。

不同题目的初始化不同：

- 最长公共子序列：空串和任何串的答案都是 `0`
- 编辑距离：空串变成长度为 `j` 的串，需要 `j` 步
- 方案数问题：空串有时对应 `1` 种方式，有时对应 `0` 种方式

所以初始化不能死记，必须从`状态定义`反推出去。

### 4. 一个简洁的思维口令

做双序列动态规划时，可以先在脑子里过这四句话：

- `定前缀`
- `看末尾`
- `分匹配`
- `设边界`

这四步走顺了，大多数二维字符串动态规划题都能找到方向。

## 二、前缀的定义和理解

### 1. 什么是前缀

在字符串或数组中，`前缀`就是从开头开始，截取前面一段连续的部分。

例如字符串 `"abcde"` 的前缀有：

- `""`
- `"a"`
- `"ab"`
- `"abc"`
- `"abcd"`
- `"abcde"`

如果说“前 `i` 个字符”，意思就是下标范围：

```text
[0, i - 1]
```

### 2. 为什么动态规划喜欢用前缀

因为前缀有三个特别适合动态规划的特点：

- `边界清楚`：前 `0` 个元素就是空序列，初始化天然明确。
- `规模统一`：前 `i` 个到前 `i - 1` 个，只减少了一个元素。
- `顺序保留`：前缀不会打乱原序列中的相对顺序。

这会让状态定义非常稳定，也会让状态转移更容易围绕“最后新增的一个元素”展开。

### 3. 怎么理解“前缀问题”

所谓“前缀问题”，就是把原问题缩小成：

- 只看第一个序列前 `i` 个元素
- 只看第二个序列前 `j` 个元素
- 问这时候的答案是什么

这种缩小后的问题，和原问题形式是一样的，只是规模更小，因此很适合作为动态规划的子问题。

### 4. 前缀和“最后一个元素”的关系

前缀之所以好用，是因为：

- 前 `i` 个元素 和 前 `i - 1` 个元素 的区别，只在最后一个元素

所以一旦状态定义成前缀，我们在推导当前状态时，往往只需要思考：

- 最后一个元素要不要参与答案
- 如果参与，它和另一个序列的末尾元素是什么关系
- 如果不参与，就退回到更小的前缀问题

这就是很多动态规划转移都围绕“最后一个字符”展开的原因。

## 三、双序列动态规划和前缀之间的关系

### 1. 为什么双序列动态规划天然适合前缀定义

双序列动态规划研究的是两个序列之间的关系，而这类关系通常只依赖：

- 元素的相对顺序
- 元素之间能否匹配、能否转化、能否组合

前缀天然保留顺序，因此非常适合拿来定义子问题。

也就是说，把原问题写成：

```text
dp[i][j] = 第一个序列前 i 个元素 和 第二个序列前 j 个元素 的答案
```

有两个直接好处：

- 规模变小后，问题形式不变
- 当前状态只比更小状态多出两个末尾元素

这就让状态转移变得很自然。

### 2. 为什么最长公共子序列和前缀特别契合

`最长公共子序列` 只要求：

- 挑出的字符顺序一致
- 不要求连续

这意味着：

- 你可以在前缀中自由决定某个字符选还是不选
- 去掉末尾一个字符后，剩下的仍然是同类问题

因此，最长公共子序列特别适合定义为：

```text
dp[i][j] = text1 前 i 个字符 和 text2 前 j 个字符 的最长公共子序列长度
```

然后只看两个末尾字符：

- 如果相等，说明它们可以一起作为公共子序列的最后一个字符
- 如果不相等，说明至少有一个末尾字符不能同时出现在当前最优解中

于是状态转移就很自然：

```text
如果 text1[i - 1] == text2[j - 1]
    dp[i][j] = dp[i - 1][j - 1] + 1
否则
    dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
```

### 3. 这句“前缀很适合”到底是什么意思

“最长公共子序列本身就和前缀很适合”这句话，实际表达的是下面三层意思：

- `问题缩小后形式不变`：原问题和前缀子问题仍然都是“求两个序列的最长公共子序列”。
- `状态容易递推`：前缀只比更小前缀多一个末尾元素，方便分析新增信息。
- `顺序天然保留`：子序列只关心顺序，而前缀恰好不破坏顺序。

所以，前缀不是随便选的一种定义方式，而是最符合这类题结构的一种状态表达。

### 4. 用一个例子理解这种关系

设：

- `text1 = "abcde"`
- `text2 = "ace"`

如果要求：

```text
dp[5][3]
```

它表示的是：

- `"abcde"` 和 `"ace"` 的最长公共子序列长度

当我们研究这个状态时，只需要看末尾字符：

- `text1[4] = 'e'`
- `text2[2] = 'e'`

它们相等，因此：

```text
dp[5][3] = dp[4][2] + 1
```

这里的 `dp[4][2]` 表示：

- `"abcd"` 和 `"ac"` 的最长公共子序列长度

你会发现，大问题缩成了更小的前缀问题，但问题本质一点没变，这正是动态规划最喜欢的结构。

## 四、双序列动态规划的常见题目整理

### 1. 最常见的题型分类

双序列动态规划常见可以分成下面几类：

#### 1. 匹配长度类

- `1143. 最长公共子序列`
- `718. 最长重复子数组`

这类题通常求的是：

- 最长能匹配多长
- 最长公共部分有多长

#### 2. 转换操作类

- `72. 编辑距离`
- `583. 两个字符串的删除操作`

这类题通常求的是：

- 从一个序列变成另一个序列，最少需要多少步

#### 3. 方案统计类

- `115. 不同的子序列`

这类题通常求的是：

- 某种匹配方式一共有多少种

#### 4. 可行性判断类

- `392. 判断子序列`

这类题通常求的是：

- 能不能匹配
- 是否满足条件

### 2. 经典题目对照表

| 题目 | 状态定义 | 转移特点 |
| --- | --- | --- |
| `1143. 最长公共子序列` | `dp[i][j]` 表示两个前缀的最长公共子序列长度 | 相等取 `左上 + 1`，不等取 `上/左` 最大值 |
| `72. 编辑距离` | `dp[i][j]` 表示前缀之间的最少编辑次数 | 不等时考虑插入、删除、替换 |
| `115. 不同的子序列` | `dp[i][j]` 表示前缀之间形成目标串的方案数 | 相等时可选可不选，不等时只能跳过 |
| `583. 两个字符串的删除操作` | `dp[i][j]` 表示前缀变相等的最少删除次数 | 常与编辑距离、LCS 相互关联 |
| `392. 判断子序列` | `dp[i][j]` 表示前缀之间是否能匹配或匹配长度 | 也可以用双指针，但 `dp` 思路相同 |
| `718. 最长重复子数组` | `dp[i][j]` 表示以两个末尾元素结尾的最长公共连续长度 | 和 LCS 不同，它要求连续 |

### 3. 做题时的固定识别方式

如果一道题同时出现以下特征，就可以优先往双序列动态规划上想：

- 题目给了两个字符串或两个数组
- 题目在问它们之间的匹配关系、转换关系或统计关系
- 题目要求最长、最短、方案数、是否可行
- 题目明显和“顺序”有关

这时可以优先尝试写出：

```text
dp[i][j] = 两个前缀的某种答案
```

然后继续问自己三个问题：

- 当前只新增了哪两个末尾元素
- 这两个末尾元素能不能发生关系
- 如果不能同时使用，应该舍弃谁

这里的“ 匹配关系 、 转换关系 、 统计关系 ”，其实是在帮助你判断：

- 这道题到底在研究两个序列之间的哪一种联系
- dp[i][j] 应该表示什么类型的答案
你可以把它们理解成双序列动态规划里最常见的三种提问方式。

匹配关系

- 匹配关系 关注的是：两个序列里，哪些元素能够对应上，能够一起组成某种结果。
- 这类题通常不是问“改多少次”，而是问“能对上多少”“能不能对上”“最长能对上多长”。

常见题目描述会出现这些说法：

- 两个字符串的 最长公共子序列
- 两个数组的 最长重复子数组
- 判断一个字符串是否是另一个字符串的 子序列
- 两个字符串中有多少字符可以按顺序对应上

这类题的核心是在问：

- 两个序列之间， 怎么对齐
- 某两个位置上的元素， 能不能配成一对
- 如果能配对，是否能接到前面的答案后面

比如 1143. 最长公共子序列 ：

- 字符相等，就可以匹配
- 字符不等，就不能同时作为当前匹配结果的一部分


所以“匹配关系”本质是在看：

- 能不能对应
- 对应后能形成多大的结果

转换关系

- 转换关系 关注的是：能不能通过一些操作，把一个序列变成另一个序列。
- 这类题通常会给你一组允许的操作，然后问最少需要几步，或者代价是多少。

常见题目描述会出现这些说法：

- 把字符串 word1 转换成 word2
- 允许你进行 插入、删除、替换
- 使两个字符串相同，最少删除多少次
- 经过若干操作后，使两个序列满足某种条件

这类题的核心是在问：

- 当前两个末尾元素是否已经一致
- 如果不一致，要通过哪种操作把问题变小
- 哪种操作代价最小

比如 72. 编辑距离 ：

- 如果末尾字符相等，不需要操作
- 如果不相等，就要考虑：
  - 删除
  - 插入
  - 替换

所以“转换关系”本质是在看：

- 怎么从序列A变成序列B
- 变化过程的代价是多少

统计关系

- 统计关系 关注的是：满足某种匹配条件的方案一共有多少种。
- 这类题不是问“最长”或“最少”，而是问“有几种方法”。

常见题目描述会出现这些说法：

- 有多少种不同的方式
- 方案数是多少
- 从一个字符串中删掉一些字符，得到目标串的方案数
- 满足条件的子序列个数

这类题的核心是在问：

- 当前两个位置上，如果字符相等，是否有“选”和“不选”两种可能
- 如果字符不相等，是不是只能跳过某一个字符
- 不同决策带来的方案数如何累加

比如 115. 不同的子序列 ：

- 如果当前字符相等：
  - 可以用这个字符去匹配目标字符
  - 也可以不用这个字符
- 所以要把两部分方案数加起来

所以“统计关系”本质是在看：

- 满足条件的路径有多少条
- 不同选择的结果怎么累加

三者的区别 你可以这样区分：

- 匹配关系 ：问的是“能不能对上，对上多少”
- 转换关系 ：问的是“怎么变过去，代价多大”
- 统计关系 ：问的是“满足条件的方式有多少种”

它们对应的答案类型通常也不一样：

- 匹配关系 ：通常是最长长度、布尔值
- 转换关系 ：通常是最小操作数、最小代价
- 统计关系 ：通常是方案数

拿一句题目描述怎么判断 你做题时可以直接抓关键词。

如果题目里出现这些词，通常偏向 匹配关系 ：

- 最长公共
- 子序列
- 是否匹配
- 是否为子序列
- 最长重复

如果出现这些词，通常偏向 转换关系 ：

- 转换
- 编辑
- 删除
- 插入
- 替换
- 最少操作

如果出现这些词，通常偏向 统计关系 ：

- 多少种
- 方案数
- 个数
- 不同的子序列
- ways / count

举个对照例子 同样是两个字符串，不同问法就对应不同关系：

- “它们的最长公共子序列长度是多少？”
  
  - 这是 匹配关系
- “把 word1 变成 word2 最少要几步？”
  
  - 这是 转换关系
- “从 s 中删字符得到 t 一共有多少种删法？”
  
  - 这是 统计关系

你会发现：

- 对象还是两个字符串
- 但问题关注点完全不同
- 所以 dp[i][j] 的含义也要跟着变

和状态定义的对应关系 这三类关系通常会把 dp[i][j] 定义成三种不同东西：

- 匹配关系 ：
  
  - dp[i][j] = 两个前缀的最长匹配长度 / 是否能匹配
- 转换关系 ：
  
  - dp[i][j] = 一个前缀变成另一个前缀的最小操作数
- 统计关系 ：
  
  - dp[i][j] = 一个前缀形成另一个前缀的方案数

所以你看到题目后，先不要急着想公式，先问自己：

- 题目是在问“对上多少”？
- 还是在问“变过去要几步”？
- 还是在问“有多少种做法”？

这个问题一旦答出来，状态定义通常就顺了。

一句话总结

- 匹配关系 ：研究两个序列怎么对应
- 转换关系 ：研究一个序列怎么变成另一个
- 统计关系 ：研究满足条件的对应方式有多少种

### 4. 学习建议

学习双序列动态规划时，推荐按这个顺序练习：

1. `1143. 最长公共子序列`
2. `392. 判断子序列`
3. `72. 编辑距离`
4. `583. 两个字符串的删除操作`
5. `115. 不同的子序列`
6. `718. 最长重复子数组`

这样安排的原因是：

- 先用 `1143` 理解最经典的二维前缀状态定义
- 再逐步接触判断类、最小操作类、方案数类和连续匹配类
- 最后形成完整的双序列动态规划题感

## 五、总结

- `双序列动态规划` 的本质，是研究两个序列之间的关系，并用二维状态记录前缀问题的答案。
- `前缀` 的本质，是从开头截取一段连续部分，它边界清楚、规模统一、顺序保留，非常适合作为动态规划状态。
- `双序列动态规划和前缀之间的关系`，本质上是：前缀能把大问题稳定地拆成更小的同类问题，并且让转移只围绕末尾元素展开。
- `最长公共子序列` 是理解这套思维最经典的一题，因为它完整体现了“定义前缀状态、观察末尾字符、分类讨论转移”的全过程。

可以把双序列动态规划记成一句话：

```text
先定义两个前缀的状态，再观察两个末尾元素如何影响答案
```
